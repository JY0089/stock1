# 내 애플리케이션 상세 관리 규정 문서

네이버 오픈 API를 이용하려면 필수적으로 네이버 개발자 센터 시스템 내부에 당신의 프로젝트 환경을 대변할 '애플리케이션' 단위를 정의해야 합니다. 이 문서는 생성을 넘어 애플리케이션 환경을 수정하고 통계 및 테스터를 디테일하게 설정하여 실무 환경에 활용하는 종합 관리 화면 기능 명세입니다.

개발자 센터 홈페이지 메뉴 트리 중 [Application > 내 애플리케이션] (주소: https://developers.naver.com/apps/#/list) 라우팅을 클릭하시면 개발자가 현재 오너 권한을 가지고 등록한 애플리케이션의 앨범과 같은 전체 목록 세트를 명확하게 확인할 수 있습니다. 목록 리스트 상단 인터페이스 중앙에 위치해 있는 'Application 등록' 버튼을 누르시면 새로운 프로젝트 용도 애플리케이션을 새로이 추가 및 론칭할 수 있습니다. 단, 어뷰징 생성을 방지하기 위해서 한 계정당 등록할 수 있는 최대 애플리케이션 숫자는 물리적으로 **10개**까지만 제한하여 부여됩니다. 더 많은 앱이 필요할 경우 기존의 사용하지 않는 유휴 애플리케이션을 폐기 삭제해야 합니다.
모든 앱이 등록된 상태에서 애플리케이션의 이름을 개별적으로 클릭하거나, 좌측 네비게이션 트리 하위에서 접근하시면 다각도에서 관리 가능한 패널로 화면이 갱신됩니다.

## 1. 기본 정보 열람 (개요 탭)
관리 화면 상단의 메뉴에서 가장 먼저 띄워지는 '개요 탭'을 클릭하시면 전체적인 개발 현황이 노출되며 기초적이고 핵심적인 정보 조회가 가능합니다.
- **애플리케이션 정보**: 클라이언트 아이디 (Client ID)와 클라이언트 시크릿 (Client Secret)의 값을 육안으로 확인할 수 있는 공간입니다. Client Secret은 보안을 위해 마스킹 처리되어 있으며 보기 버튼을 눌러야 온전한 문자열 획득이 가능합니다. 이 키값들이 유출될 경우 즉각 키를 갱신해야 합니다.
- **API 호출 안내**: 초심자를 위해 API 호출 과정에서 발생하는 에러 코드가 무엇인지 진단 시트 및 트러블슈팅 가이드를 하이퍼링크 형태로 제공하여 이슈 대응력을 높입니다.
- **비로그인 오픈 API 당일 사용량 막대 차트**: 검색, 데이터랩 등 실시간 단발 호출 위주의 현재 사용량이 매일 배정되는 트래픽 한도에서 얼마나 점유율을 차지하고 막대를 채웠는지 그래프 상태를 시각적으로 보여줍니다. 여기서 한계치 도달을 미리 파악할 수 있으므로 상용화 중에 중요한 모니터링 메뉴가 됩니다.

## 2. API 설정 및 변경 관리 (API 설정 탭)
최초 등록 시점 뿐 아니라, 기획 변경으로 인해 호출할 API 명세가 바뀌거나 앱 아이콘 등이 개선될 경우 이곳에서 대폭 수정이 이뤄집니다.
- **카테고리 정보**: 이 앱이 엔터테인먼트, 금융, 유틸리티 등 구체적으로 어떤 서비스 비즈니스 분야에서 작동하는지 분류 항목을 명시적으로 재조정할 수 있습니다. 이는 사용자의 권한 동의 창에 표시되기도 합니다.
- **로고 이미지 파일 등록**: 네이버 소셜 로그인이나 권한 획득 등을 통해 고객사 시스템으로 포워딩되는 과정에 띄워주는 자사의 브랜드 스퀘어 이미지입니다. 권장되는 이미지 픽셀 크기에 맞춰 파일을 업로드, 적용하시면 됩니다. 변경 시 시스템 캐시로 인해 곧바로 확인이 어려울 수도 있으니 유의하시기 바랍니다.
- **애플리케이션 개발 상태 스위칭 규칙**: 이게 오픈 API 관측의 가장 중대한 사안입니다. '로그인 방식 오픈 API (OAuth)' 체계를 사용하는 애플리케이션들은 보안 검수라는 관문을 통과하기 전에는 본질적으로 최초 상태인 **개발 중 (In Progress)** 상태로 머물게 됩니다. 개발 중 상태가 유지되는 한 기획과 개발 과정 동안에는 외부 일반 유저가 아니라, 오로지 이 메뉴 하단의 멤버 관리 탭에서 [테스터 ID] 로 등록을 마친 20개의 한정된 네이버 아이디들 만이 인증 로그인 환경을 정상적으로 작동시킬 권한을 부여받습니다.
이 과정의 철저한 테스트가 끝나서 실제 프로덕션 서버에 애플리케이션을 상용 적용하기 원한다면 필수적으로 공식적인 '네이버 로그인 검수요청' 기능을 작성하셔서 제출 평가를 거치셔야 합니다. 이 과정에서 화면 캡처, 가이드 위반 여부 등이 종합 체크되며, 이 절차가 완수되어 합격 통보(승인 인증)를 받으면 마침내 개발 상태가 **서비스 적용 (Deployed)** 이라는 영광스런 상태 지표로 교체 처리되어집니다. 비로소 이 상태에서야 테스터 계정 리스트에 제한받지 않고 불특정 무한 다수의 실존 네이버 회원들이 로그인 및 데이터 연계 시도를 자유로이 할 수 있게 개방됩니다. 네이버의 관련 추가 문서인 '검수 요청 등록 가이드'를 통해 필수적으로 합격 가이드라인 숙지가 요구됩니다.

## 3. 멤버(팀) 운용 관련 자원 등록 (멤버 관리 탭)
이 프로젝트의 성장에 따라서 혼자만이 아닌 동료 관리자와 외부 검증 테스터 조직을 구성할 때 필요합니다.
- **관리자 ID 등록**: 최상위 개발자가 아닌 추가적으로 애플리케이션 콘솔 환경을 둘러보고 관리 모니터링을 도울 부 관리자 인원의 실명 네이버 아이디를 3개까지만 제한적으로 락온(Lock-On)하여 수록할 수 있습니다.
- **테스터 ID 등록**: 앞에서 언급했던, 개발 상태가 실 서비스가 아닌 '개발 중' 모드일 때 네이버 로그인 세션 호출 및 허용 실험이 가능한 테스트 계정용 아이디 풀(Pool) 창구입니다. 개발자 본인 이외에 최대 20개의 네이버 아이디를 인서트(Insert) 할 수 있어 QA 검증 작업에 유리합니다.

## 4. 트래픽 운영 통계 확인 뷰
현재 배포된 클라이언트 시스템이나 외부 웹 사이트가 API를 얼마나 폭발적으로 사용하고 있는지 일일 점검을 하기 위한 애널리틱스 영역입니다.
- **로그인 통계 대시보드**: 애플리케이션에 순수 네이버 연동 로그인을 시도, 그리고 성공한 사용자의 트래픽 흐름 통계를 그래픽으로 보여 줍니다.
- **API 통계 대시보드**: 검색 API나 카페 등 다른 단위 API의 트래픽을 일별/월별, 성공/에러 등 특정 기간에 기반을 두어 트렌드별 통계 확인을 할 수 있습니다. 앱 유지보수에 가장 귀한 수치적 증거입니다.

## 5. 최종 데이터 파기 및 애플리케이션 삭제
가장 조심스럽고도 불가역적인 최후 관리 메뉴입니다. 등록 이후 테스트 목적을 상실했거나 기업이 폐업, 또는 서비스를 온전하게 갈아엎을 때 애플리케이션 자체를 지웁니다.
이 작업이 수행되는 즉시 사용자들의 기존 획득 토큰은 셧다운 처리되어 고객들의 네이버 로그인이 모두 오류를 토해내며, 통계 지표 정보 등도 완벽한 소거 과정을 통해 복원 롤백이 불가해집니다.
명확하고 조심스런 삭제 액션 프로세스는 아래와 같습니다:
1단계: 제거하기를 희망하는 대안 애플리케이션의 관리 페이지 뷰에서 'API 설정' 탭을 진입해 레이아웃 하단 스크롤 영역을 확인.
2단계: 경고 문구와 함께 있는 빨간색 텍스트 계열의 '애플리케이션 삭제의 애플리케이션을 삭제합니다' 라는 확인 버튼을 재차 정교하고 뚜렷하게 클릭하게 하여 물리적 삭제를 실행 처리합니다.

# 오픈 API 생태계 용어 해석 대백과

네이버는 방대한 기능을 열어두고 수많은 기업 및 개발자와 소통하기 위해 이 오픈 API라는 시스템을 도입하였습니다만, 각기 다른 베이스 라인을 가진 개발 환경과 업계 관례의 차이로 인해 오픈 API 생태계와 결부된 기술적 단어들이 사용자마다 너무나 다양한 의미로 제멋대로 해석, 사용되는 문제점을 파악하게 되었습니다. 처음 서버 개발이나 프론트 연동 셋업을 시작하는 주니어 개발자는 물론이고, 타 시스템의 레거시 구조에만 치중해왔던 시니어 개발자에게까지 문서 속 용어의 오해석이 자칫 데이터 연동 구현 로직의 치명적 결함과 혼란이라는 나비효과를 주기도 합니다.
이에 따라 네이버의 관련 기술 부서는, 그 어떠한 프로그래머라도 절대적이고 획일화된 개념을 가지고 오픈 API를 오차 없이 다룰 수 돕게끔, 네이버 개발자센터 내에 고유하게 규칙화되어 통용되는 전문 용어 개념들과 패러다임을 한 곳에 매우 구체적이고 꼼꼼하게 정의 및 나열하여 가이드합니다.

## 제1장: API 구성의 기본 개념 이해

### 1-1. API (Application Programming Interface)
가장 근원적인 의미로써, 프로그래머나 운영체제 혹은 광범위한 시스템 아키텍처 라이브러리 등이 내부 동작 방식에 관여하지 않고 오로지 제공되는 접점 규격(인터페이스)을 이용 및 호출함으로써 독자적인 응용 프로그램 소프트웨어를 순조롭게 작성하는 방식을 일컫는 단어입니다. 운영 체제가 자체적으로 제공하는 복잡한 호출을 추상화로 간단히 한 Window 대상의 API, Java 환경의 API, 크롬 등의 브라우저에서 카메라나 스토리지에 관여하는 최신 HTML5 API 모듈, 모바일 폰 센서 제어용 Android API 등 셀 수 없이 다종다양하게 포진해 있습니다.

### 1-2. 오픈 API (Open Application Programming Interface)
수많은 내부 폐쇄형 인터페이스인 API 중에서 거대 플랫폼(네이버 등)의 핵심 자원들, 즉 검색의 결과 처리 기능, 캡차 이미지 제공 기능 또는 방대한 블로그 포스팅 데이터 콘텐츠 자체를 폐쇄된 자체 서버 안에서만 돌리는 것에 안주하지 않고, 회사 외부의 불특정 다수 대중 개발자가 통상적으로 널리 쓰이는 표준적인 웹 프로토콜 통신인 HTTP 네트워크 통신망 환경으로 누구나 데이터를 직접 호출 조회해 이용할 수 있게 시스템 벽을 외부에 열어젖히고 완전하게 '개방(Open)' 해버린 인터페이스 아키텍처를 뜻합니다. 네이버 개발자 오픈 센터 룸에서 제공, 설명하는 단어 번역, 클라우드 얼굴 검색 데이터, 최신 캡차 방어 등 절대 다수의 API는 바로 이 HTTP 통신 요청 전송 기법으로 이룩된 완전한 오픈 API 성향을 가집니다.

### 1-3. 모형 애플리케이션 (Application)
네이버가 만들어 공개한 시스템이 원자재라면 이 원자재라는 '오픈 API'를 조합하고 스크립트에 포함 시켜 당신의 손에서 탄생하게 된 다양한 최종 소비 프로그램이나 패키지 및 결과물을 통칭해 바로 '애플리케이션'이라고 정의합니다. 이 결과물은 광활한 모바일 앱이 될 수도 있고, 데스크톱 소프트웨어가 될 수도, 또는 PC 브라우저 내의 웹 페이지나 플러그인, 또는 모바일 웹 시스템 등 형태에 한계를 두지 않습니다. 이 네이버 오픈API 기능을 이용할 자격을 얻으려면 우선 네이버의 개발자센터 사이트 안 '애플리케이션 등록'이라는 서브 메뉴에 본인의 결과물 정보와 콜백 도메인, 구동 환경 등을 명세 폼에 입력 등록함으로써 자격을 갖춘 인스턴스를 하나 발급, 등록받아야만 합니다.

## 제2장: API 데이터 무결성 입증 및 인증 (Authentication) 파트

### 2-1. API 게이트웨이 (API Gateway)
클라이언트의 요청이 도달하는 첫 관문의 방어자입니다. 현재 네이버에서 퍼블리시되는 대다수의 오픈 API 시스템은 물리적, 논리적 구조적으로 단일 창구인 고정 호스트 `https://openapi.naver.com/` 최상단 엔드포인트 서버를 무조건적으로 통과하며 내부의 알 수 없는 복잡한 진짜 프로세서 API 서버들로 라우팅 됩니다. 바로 이 최전선 역할을 수행하는 통합 관제 서버를 'API 게이트웨이' 서버로 칭합니다.
게이트웨이가 담당하는 주 로직은 단단합니다. 방대한 인터넷상 누군가 오픈 API 자원 호출을 던지는 순간 이 게이트웨이가 요청 헤더와 주소를 먼저 받아들인 뒤, 이 호출 양식이 맞는지 1차 스캐닝을 하고, 해당 클라이언트 ID가 실제 가입 후 권한이 인증된 합법적 사용자의 호출이 맞는지를 이중 체크하며, 시간당 할당된 트래픽 기준에서 할당된 호출 허용량을 과도하게 초과한 무한 크롤링 시도인 건 아닌지 등을 순식간에 복합 스크리닝 합니다. 이 모든 엄중한 스크린이 정상 합격을 판정받으면 그때 비로소 숨겨진 실제 API 코어 전용 서버 구동 결과를 외부인에게 꺼내 주고 반환시켜 넘깁니다. 단 하나라도 어긋난 비정상 접근의 경우 서버 통과를 배제하고 곧장 오류 상황 코드와 적나라한 오류 발생 이유의 메시지를 프론트로 뱉어냅니다.

### 2-2. API 키 시스템의 몰락 또는 레거시 키
이 문단은 과거의 유산을 청산하기 위한 안내입니다. 2016년 1월 26일이라는 기준 연도라는 시간 이전에는, 단순히 어떤 API든지 이용 승인을 받을 경우 무작위 문자열 하나, 즉 이른바 '단일 API 키'라는 것만을 생성해 발급해주곤 했습니다. 이 키는 당신이 진짜로 합법적 인증 회원임을 입증할 유일할 티켓이었습니다. 호출 때마다 이걸 헤더에 붙여 게이트웨이 서버의 승인을 받아냈었습니다. 하지만 시대가 변하며 2016년 1월 26일 시점 이후 전면적으로 시스템이 개편되었고, 이 단순 취약했던 구식 API 전용 키 형식은 더는 인증 파이프라인에서 권장 및 사용되지 않습니다. 이를 완벽하게 대신하고자 현재는 ID와 Secret이라는 한층 더 보안이 강화된 양방향 방식을 사용합니다.

### 2-3. 클라이언트 아이디(Client Id)와 클라이언트 시크릿(Client Secret)
이 인증 쌍은 네이버 생태계를 돌아다니며 현재 권한을 행사하는 주체가 허가받은 정확한 고객인지 여부를 최종 크로스 체크하는 고유한 세트 아이디 영문자열과 시크릿 비밀번호 조합값입니다. 네이버 개발자센터 공간의 애플리케이션 생성 관리 등록 메뉴얼을 따라 앱을 등록(Register) 완료하는 그 찰나의 순간에 단 1번만 콘솔에 생성, 부여 제공되는 특급 민감 보안 데이터 값입니다. 외부 유출 방지에 힘쓰십시오.

## 제3장: 데이터 호출 전송 (HTTP Call) 프로세스

### 3-1. 메인 요청 URL (Request URL Endpoint)
서버에 위치한 오픈 API 프로세스를 실제 구동시키고 발동시키기 위해서, 내가 이 위치의 코드를 깨운다는 것을 명시적으로 알리는 인터넷상의 정확한 리소스 웹 주소(URL) 좌표를 일컫습니다.
앞단에서 언급된 체계 변경에 의해 과거 http://openapi.naver.com 형태만 쓰던 것에서 진일보하여, 현재는 모든 호출이 https 보안 채널 하에 이뤄지며 주소 또한 계층 구조를 지닙니다. 오늘날 당신의 코드가 가리켜야 할 URL 구조는 `https://openapi.naver.com/[버전정보]/[서비스구분]/[구체적 API기능구분]` 형태와 같이 매우 시멘틱하게 변경 조립되어 사용하도록 설계되어 서비스 중입니다.

### 3-2. 요구 요청 변수 (Request Parameter Elements)
호출 행동을 취하며 API 게이트웨이 서버를 향해 함께 무거운 보따리로 던져 전송해야만 하는 제어용 세부 데이터 값들의 묶음 집합입니다.
예를 들면 블로그 검색 시 '무엇을 찾고 싶은지', '총 몇 개의 리스트를 가져올 건지'가 바로 요청 변수입니다. API 시스템 레퍼런스를 면밀하게 읽고 참고하시어 내가 짠 코드에서 보내야만 하는 변수의 영문 스펠링 이름이 틀려 인식 불가 판정이 나거나, 필수로 요구되는 중요 규약 변수 텍스트가 깜빡하고 누락되어 404/400 통신 반려 응답을 받지 않도록 각별히 코딩 시 주의 점검에 힘쓰셔야 합니다.

### 3-3. URL 인코딩 통번역 (URL Encoding Scheme)
인터넷 익스플로러와 웹 브라우저가 다루는 고유 체계에 관한 것입니다. 우리가 보내고자 하는 검색 단어 파라미터 값 그 중간 어딘가에 순수 한글 형태나 심볼마크 특수 문자가 들어간 채로 무방비하게 텍스트 그대로 인터넷 선망 서버로 그냥 던져버린다면, 글자가 와전되거나 박살, 혹은 흔히 말하는 '깨지는' 비극이 초래됩니다. 그렇기 때문에 데이터를 온전히 서버에 안착하게 하기 위해 국제망의 지정된 변환 규칙표 체계에 따라서 이 한글 문잣값을 기계가 이해하는 비트적 영문, `%20`과 같은 퍼센트 인트 특수 코드 텍스트 덩어리로 변환하는 전 단계 처리를 선행해야만 무사 전송됩니다. 이 귀찮지만 아름다운 변환 과정을 IT 업계에선 흔히 'URL 인코딩(URL Encoding)' 작업이라 지칭합니다. 역으로 전송이 무사히 다 끝나고 결과값으로 돌아온 저 괴상한 `%E2` 덩어리들을 분석해 다시 인간이 읽을 수 있는 멀쩡한 문자로 재해석 변환 복구시키는 행위 전반을 'URL 디코딩(URL Decoding)'이라고 멋있게 부릅니다.

## 제4장: 네트워크 통신(HTTP)의 백본 이해

### 4-1. HTTP 메서드 (Methodology)
이 부분은 웹 통신 자체의 약속입니다. 월드 와이드 웹의 HTTP 프로토콜 규격 구조에서 당신의 프론트 웹 서버가 백 서버로 요청 데이터, 그리고 그에 응하는 응답 전송 데이터를 막 주고받을 때 사전에 기계들끼리는 이러이러하게 동작해달라고 암묵적으로 약속으로 묶어둔 깃발 체제 동작 방식입니다. 국제 웹 표준을 이끌어가는 조직 W3C 규약에서는 데이터를 요구하기만 하는 무설정의 `GET`과 데이터를 무겁게 밀어 넣어 생성하는 `POST`, 교체하는 PUT 등 약 8가지 넘는 메서드 문자를 표준 정의시켜놓고 있습니다. 따라서 개발자가 네이버의 해당 특정 오픈 API 모듈 서버를 향해 올바른 요청을 던지기 위해서는, 안내서 레퍼런스 페이지에 이 API를 가져갈 땐 무조건 어느 한 메서드 타입 형식만을 사용하라 지정해둔 엄격한 강제 규칙(일반 검색의 GET, 캡차 제출의 POST 등)에 반드시 맞추어서만 코드를 작성해야 성공적으로 데이터 교착이 해결됩니다.

### 4-2. HTTP 헤더 (Header Payload)
무언가를 담는 거대한 택배 상자 겉에 부착된 송장 상세 라벨이 바로 HTTP 헤더 구조라고 보면 됩니다. 이 헤더 포맷 내부에는 실제 웹 서버 쪽으로 던져보내는 호출의 순수 내용과는 별개로 이 데이터 세트의 메타적인 정보를 구구절절 설명하는 서브 정보들의 키 세트가 알차게 묻혀 있습니다. 요약하자면 현재 수행하는 메서드가 GET 인지 무엇인지 하는 정보와, 정확히 가리키려 하는 목적 점 요청 패스 대상 URL이 어디인지 나타내는 정보, 그리고 통신에 쓰이는 HTTP 프로토콜 버전 넘버 등이 빼곡히 기록되어 안착합니다. 여기에 우리가 자유 재량으로 추가적인 약속된 이름 정보와 그에 따른 밸류 키 값을 수기 작성하여 전송 탑재할 권리가 주어집니다. 따라서 개발자가 네이버 오픈API 구조체를 이용하려 할 때는 앞서 발급받았던 애플리케이션 권한 인자 즉, '클라이언트 아이디' 문자열과 '클라이언트 시크릿' 패스워드 텍스트 덩어리를 무조건 반드시 이 헤더 상자 내부에 단단히 봉인, 기본적으로 포함하여 게이트웨이로 전송해 통과해야만 허가가 납니다.

### 4-3. HTTP 응답 상태 코드 (Status Response Code Numeric)
내 컴퓨터에서 웹 서버를 향해 성공적으로 어떠한 API 요청 전송 완료하고 나면 웹 서버는 묵묵부답으로 있지 않고 이 API 호출 및 처리 과정 절차가 정상적으로 무사히 처리 완료되었는지, 아니면 과정 중 어떤 심각한 오류가 초래됐는지에 관련된 응답 정보를 명확한 3자리 숫자의 HTTP 상태 코드 값(Number)과 패키징하여 무조건 나에게 다시 포장하여 반환해 돌려줍니다.
일례를 들자면 우리가 수행한 호출 명령이 완벽에 가깝게 성공적으로 이뤄졌다고 서버가 판독했을 때에는 만족의 의미로 상태 코드의 값으로 '200' 이라는 숫자를 반환합니다. 반면 API를 구동시키는 우리의 방식이나 파라미터가 치명적으로 위배 또는 문법 오류가 발생했을 때에는 에러 코드 체계인 400(잘못된 문법), 403(권한 절대 거부 오류), 404(찾을 수 없는 공허한 페이지) 등의 코드가 반환되곤 합니다. 더 나아가 네이버 인프라 서버 자체가 오류가 생겼을 때는 500 내장 코드를 던져줍니다. 이러한 네이버 오픈 API 고유 호출 시 발생하는 여러 상태 응답 코드와 고유 이유의 해석 정보는 별도 페이지인 "오류 코드 정리 가이드" 문서를 면밀히 참고하시어 디버깅 전략에 활용하십시오.

### 4-4. 인증과 OAuth 2.0 (Security & Protocol Framework)
인증(Authentication) 메커니즘은 사이버 공간에서 대체 어떤 사용자가 화면 너머의 본인 진실 여부를 판단하고 식별하는 절대적인 규율 잣대입니다. 네이버에 로그인할 때 네이버 홈 창구 회원이 네이버 실 아이디와 암호를 수기로 키보드로 입력해서 접속 성공하는 흐름 과정 그 자체라고 생각하는 게 편합니다.
그러나 이런 상식도 문제가 될 때가 있습니다. 네이버 직원이 아닌 외부 프로그래머가 네이버의 광범위한 회원 인프라 정보를 추출하려 서드파티 앱을 만들었을 때, 당신의 소중한 '네이버 실 아이디와 원본 암호' 자체를 남이 짠 사설 보안 화면에 입력한다면 프라이버시 침해 위험입니다.
이런 불완전한 상황에서는 오직 검증된 네이버 보안망 자체 공식 시스템 창구가 튀어나와 사용자 인증 판정을 독립적으로 한 뒤, 그 자격으로 인증 완료되었다는 '티켓' 데이터만을 외부 애플리케이션으로 대리 중계해 리다이렉트 시킵니다. 일반 사용자는 외부 사제 솔루션을 쓰더라도 안심하고 환경을 이용할 수 있는 것입니다.
전 세계 기구가 정의한 이 완벽한 인증 승인 절차 규격이 바로 가장 유명한 'OAuth' 아키텍처라는 것입니다. 네이버의 로그인 모듈은 최신 보안 규약인 OAuth 버전 2.0을 고도화 적용하고 있습니다.

### 4-5. 접근 인증 토큰 (Access Token Value)
앞 절에서 설명한 OAuth 버전 2.0 흐름을 통해 사용자가 제3의 앱을 쓰며 로그인 권한 부여 인증 과정을 성공적으로 통과하면 네이버 플랫폼 서버는 무작위하고 복잡한 보안 문자열 티켓을 뽑아냅니다. 이 문자열의 이름이 '접근 인증 토큰(Access Token)' 이며 앱 콜백 쪽으로 최종 던져 줍니다.
이 특수하게 승인된 암호화 값 데이터는 우리가 로그인 방식 API를 호출할 때 아주 강력하게 쓰입니다. 회원 프로필 정보 조회나 카페 자동 가입 등 회원 개인 정보가 엮이는 무거운 보안 처리를 수행하는 API는 반드시 이 토큰을 필요로 합니다. 패킷 헤더 내부에 깊숙이 이 Access Token을 은닉 및 동봉하여 반드시 실어 보내야 인가가 결정됩니다.
